230 /**
231  * Represents a Guaranteed globally unique 160 bit ID within spark based on twitters 64bit snowflake id:
232  * leading 4 bits - type of id
233  * padded 4 bits for future use / ease of code
234  * next 48 bits - MAC address of the drivers host network card (unique for run with no central server / service requirement)
235  * next 32 bits - partition id
236  * next 41 bits - ms since 20210101
237  * padded 7 bits - for future use / ease of code
238  * remaining 24 bits - partition specific incremented row id - allowing 16777216 rows per ms, which is unlikely to occur
239  * and easy to manage overflow either way
240  *
241  * The organisation allows the base and first long to be fixed at the start of a partition reset with only the last long
242  * changing per row easing bit fiddling.  In the case of overflow the header remains untouched and only then must 41 bits ms value be
243  * re-evaluated, reducing clock slow downs (coincidentally around 41ms).
244  *
245  * Full time synchronisation across all clusters in a business is not required as MAC address provides driver uniqueness
246  * (assuming your cloud / network provider guarantees this uniqueness for routing).  Partition id provides segregation across
247  * a given action with timestamp ensuring repeated runs on the same cluster do not overlap.
248  */
