14 /**
15  * A collection of OSS Spark factory functions for all of Quality's runners provided in two flavours for ruleEngine, folder
16  * and expressionRunner.
17  *
18  * 1. xFactory (requires encoders for both input and the full output type)
19  * 2. xFactoryT (requires encoders for the underlying output type wrapped by Quality types)
20  *
21  * The T variants accept all spark encoders and, on later spark versions, AgnosticEncoders can also be used to specify
22  * java collections.  Not using AgnosticEncoders will typically result in cryptic messages about missing types as Java
23  * drops it's type information in generated classes.
24  *
25  * The non-T variants require using Frameless for type derivation with an import of quality.implicits._.
26  *
27  * No QueryPlan generating expressions (e.g. joins/correlated sub-queries etc.) may be used and their presence will
28  * trigger an exception.
29  *
30  * ProcessorFactory's create new instances on the basis of the compile, forceMutable and forceVarCompilation boolean
31  * parameters. Using compile = false will trigger interpreted execution of rules and fresh tree copies for each instance
32  * call.  The default compilation approach (forceMutable = false) reduces source code
33  * generation overhead for non-stateful expression trees (e.g. no monotonically_incrementing_id or unique_id or rand etc.).
34  * With forceMutable set as true a higher cost implementation based on Spark provided MutableProjections will be used,
35  * if there are issues in actual processing this option is worth trying.
36  *
37  * EXPERIMENTAL - forceVarCompilation = true uses a wholeStageCodeGen style approach for subexpressions and removes
38  * the use of InternalRow wherever possible, the default of forceVarCompilation = true uses InternalRow based processing.
39  * The compilation overhead is higher with VarCompilation, albeit typically paid once, but the runtime performance is
40  * the fastest of the options. The compilation approach however is closer to ruleRunner than WholestageCodegen and is
41  * therefor deemed supported but experimental.  It is only supported on version 3.2.1 and above.
42  *
43  * The defaults, including runner specific compilation options, are chosen for general larger volumes of data processing.
44  *
45  * In some cases the input may need a further projection, e.g. from_avro, the extraProjection function allows for this.
46  *
47  * By default the Quality optimisations are added, e.g. funRewrite.  These may be disabled vai setting
48  * enableQualityOptimisations = false.
49  */
