12 /**
13  * A collection of OSS Spark factory functions for all of Quality's runners provided in two flavours for ruleEngine, folder
14  * and expressionRunner.
15  *
16  * 1. xFactory (requires encoders for both input and the full output type)
17  * 2. xFactoryT (requires encoders for the underlying output type wrapped by Quality types)
18  *
19  * The T variants accept all spark encoders and, on later spark versions, AgnosticEncoders can also be used to specify
20  * java collections.  Not using AgnosticEncoders will typically result in cryptic messages about missing types as Java
21  * drops it's type information in generated classes.
22  *
23  * The non-T variants require using Frameless for type derivation with an import of quality.implicits._.
24  *
25  * No QueryPlan generating expressions (e.g. joins/correlated sub-queries etc.) may be used and their presence will
26  * trigger an exception.
27  *
28  * ProcessorFactory's create new instances on the basis of the compile, forceMutable and forceVarCompilation boolean
29  * parameters. Using compile = false will trigger interpreted execution of rules and fresh tree copies for each instance
30  * call.  The default compilation approach (forceMutable = false) reduces source code
31  * generation overhead for non-stateful expression trees (e.g. no monotonically_incrementing_id or unique_id or rand etc.).
32  * With forceMutable set as true a higher cost implementation based on Spark provided MutableProjections will be used,
33  * if there are issues in actual processing this option is worth trying.
34  *
35  * EXPERIMENTAL - forceVarCompilation = true uses a wholeStageCodeGen style approach for subexpressions and removes
36  * the use of InternalRow wherever possible, the default of forceVarCompilation = true uses InternalRow based processing.
37  * The compilation overhead is higher with VarCompilation, albeit typically paid once, but the runtime performance is
38  * the fastest of the options. The compilation approach however is closer to ruleRunner than WholestageCodegen and is
39  * therefor deemed supported but experimental.  It is only supported on version 3.2.1 and above.
40  *
41  * The defaults, including runner specific compilation options, are chosen for general larger volumes of data processing.
42  *
43  * In some cases the input may need a further projection, e.g. from_avro, the extraProjection function allows for this.
44  *
45  * By default the Quality optimisations are added, e.g. funRewrite.  These may be disabled vai setting
46  * enableQualityOptimisations = false.
47  */
