12 /**
13  * A collection of factory functions for all of Quality's runners provided in two flavours for ruleEngine, folder
14  * and expressionRunner.
15  *
16  * 1. xFactory (requires encoders for both input and the full output type)
17  * 2. xFactoryT (requires encoders for the underlying output type wrapped by Quality types)
18  *
19  * The T variants accept all spark encoders and, on later spark versions, AgnosticEncoders can also be used to specify
20  * java collections.  Not using AgnosticEncoders will typically result in cryptic messages about missing types as Java
21  * drops it's type information in generated classes.
22  *
23  * The non-T variants require using Frameless for type derivation with an import of quality.implicits._.
24  *
25  * No QueryPlan generating expressions (e.g. joins/correlated sub-queries etc.) may be used and their presence will
26  * trigger an exception.
27  *
28  * ProcessorFactory's create new instances on the basis of the compile and forceMutable boolean parameters. Using
29  * compile = false will trigger interpreted execution of rules and fresh tree copies for each instance call.  The default
30  * compilation approach reduces source code generation overhead for non-stateful expression trees (e.g. no
31  * monotonically_incrementing_id or unique_id or rand etc.).  With forceMutable set as true a higher cost implementation
32  * based on Spark provided MutableProjections will be used, if there are issues in actual processing this option is
33  * worth trying.
34  *
35  * The defaults, including runner specific compilation options, are chosen for general larger volumes of data processing.
36  *
37  */
